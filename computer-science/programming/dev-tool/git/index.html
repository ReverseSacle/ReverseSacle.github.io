# 概述

Git是一个免费开源的分布式版本控制系统，使用一个叫做仓库的数据库来记录文件的变化。仓库中每个文件都有一个完整的版本历史记录，可以看到谁在什么时间修改了哪些文件的哪些内容，在需要的时候也可以将文件恢复到之前的某一个版本。

有了版本控制系统，就可以跟踪每个文件的变化，让项目成员之间的协作更加高效，目前最流行的版本控制系统有两种，一种是集中式版本控制系统(例如SVＮ, CVS)，另一种是分布式版本控制系统(例如Git, Mercurial)。

  • 集中式版本控制系统 - 所有的文件都保存在中央服务器上，每个人的电脑上只保存了一个副本，当需要修改文件时，首先要从中央服务器上下载最新的版本，然后再添加新的内容，修改完成之后再上传回给中央服务器
    • 如果中央服务器出现故障或网络连接出现问题，那所有人都无法工作了，只能等待中央服务器或者网络恢复正常
  • 分布式版本控制系统 - 每个人的电脑上都有一个完整的版本库，可以在本地进行修改，不需要考虑网络问题，即使中央服务器出现故障，也可以继续工作，当需要将修改好的内容分享给其他人时，只需将仓库相互同步就行了
    • 支持离线工作以及强大的分支管理特性

# Git基础配置

在开始使用Git之前，需使用git config命令配置用户名和邮箱，只需执行一次。

  • --global - 全局配置，所有仓库生效
  • --system - 系统配置，对所有用户生效
  • 不加参数 - 表示只对当前的Git仓库有效

# Git代理监听配置

设置代理

  • 代理的本地socks5监听端口为1080

  • git config --global http.proxy 'socks5://127.0.0.1:10808'

  • git config --global https.proxy 'socks5://127.0.0.1:10808'

取消代理

  • git config --global --unset http.proxy
  • git config --global --unset https.proxy

# 新建仓库(Repository - Repo)

仓库也加版本库，可以将仓库理解为一个目录，这个目录里面所有的文件都可以被Git管理，每个文件的修改，添加，删除等操作，Git都能够跟踪到，任何时候都可以追踪历史或还原到之前的某一个版本。

有两种方式创建仓库

  • git init - 直接在本地新建一个仓库
    • git init 名称 - 当指定一个名称时，在当前目录下创建一个新的目录作为Git仓库
  • git clone "https://..xxx.git" - 从远程服务器上克隆一个已经存在的仓库

main - 当前所在分支的名称，表示当前目录已经是一个被Git管理的仓库

.git目录 - 隐藏目录，存放了Git仓库的所有数据，该目录下的文件和目录都是Git仓库的重要组成部分，不能随意删除，如果删除这个目录，该仓库也会被删除，当前所在目录就会变成一个普通的目录

du -sh .git - 查看.git文件大小

# Git的工作区域和文件状态

Git的本地数据管理分为三个区域

  • 工作区(Workong Directroy) - 也称为工作目录或本地工作目录，即当前电脑上的目录，资源管理器里能够看到的文件夹就是工作区
    • .git目录所在的当前目录
  • 暂存区(Staging Area/Index) - 也称为索引，是一种临时存储区域，用于保存即将提交到Git仓库的修改内容。暂存区是在Git进行版本控制时非常重要的一个区域
    • .git/index
  • 本地仓库(Local Repository) - 即通过git init命令等方式创建的仓库，包含了完整的项目历史和元数据，是Git存储代码和版本信息的主要位置
    • .git/objects

工作区就是实际操作的目录；暂存区就是一个中间区域，用于临时存放即将提交的修改内容；本地仓库是Git存储代码和版本信息的主要位置。

当修改完工作区的文件后，需要通过git add命令将它们添加到暂存区，然后再通过git commit命令将暂存区的修改提交到本地仓库中。

文件状态

  • 未跟踪(Untrack) - 新创建的，还没有被Git管理起来的文件
  • 未修改(Unmodified) - 已经被Git管理起来，但是文件内容没有发生变化的文件
  • 已修改(Modified) - 已经被修改过，但是还没有添加到暂存区里的文件
  • 已暂存(Staged) - 已经被修改过，并且已经添加到暂存区的文件

# 添加和提交文件

相关命令

  • git status - 查看当前仓库的状态，例如查看当前仓库所在的分支名，所包含的文件以及包含文件的状态
    • 新建文件会表示为红色，即该文件为未跟踪状态
    • git status -s - 查看当前仓库的简略状态，开头有两个?号，第一个表示暂存区的状态，第二个表示工作区的状态
  • git add - 将新创建的文件添加到暂存区
    • 之前新建文件已变为绿色
    • git add *.txt - 支持通配符来将多个文件添加到暂存区
    • git add . - 将当前文件夹中所有新建文件添加到暂存区，.表示当前目录
  • git rm --cached 文件名 - 将之前添加到暂存区的文件取回
  • git commit - 提交，只会提交暂存区中的文件
    • 提交之后，通过git status命令可见之前新建文件的信息已消失，表示之前的新建文件已提交到仓库中
    • git commit -m "提交的信息" - -m指定的信息会被记录到仓库中
    • git commit - 进入一个交互式界面，默认使用vim来编辑提交信息
    • git commit -a -m "提交的信息"或git commit -am "提交的信息" - 一个命令就可以完成添加到暂存区和提交这两个操作了
  • git ls-files - 查看暂存区的内容
  • git log - 查看提交记录，每次提交都有一个唯一的ID，具体信息包括
    • 唯一的ID - commit字段后面的16进制字符串
    • 每次提交的作者和邮箱 - 通过git config命令配置的
    • 提交的时间
    • 提交的注释信息
    • git log --oneline - 查看简洁的提交记录，只显示每次提交的ID和提交信息

可能会看到当前分支并不是main，而是master，这是由于Git的版本不同导致的。

# 回退仓库版本

git reset命令用于回退当前仓库版本，可以退回到之前的某一个提交的状态，有三种回退模式

  • git reset --soft - 回退到某一个版本，并且保留工作区和暂存区的所有已修改内容
  • git reset --hard - 回退到某一个版本，并且丢弃工作区和暂存区的所有已修改内容
  • git reset --mixed - (默认参数)回退到某一个版本，并只保留工作区的已修改内容，而丢弃暂存区的已修改内容

git reflog - 查看已执行过的操作的历史记录，开头为操作记录ID，可通过git reset --hard ID 来回退操作

# 查看比较差异

git diff命令用来查看文件在工作区，暂存区和版本库之间的差异，还可以查看文件在两个特定版本之间的差异或者文件在两个分支之间的差异。

  • git diff - 无参数，会比较工作区与暂存区之间的差异，显示发生更改的文件以及更改的详细信息
    • 当无差异时会显示空白内容
    • 第一行提示发生变更的文件
    • Git会将文件的内容使用哈希算法生成一个40位的哈希值，第二行只显示哈希值的前七位，最后的一段数字表示该文件的权限
    • 当前及以下为修改的内容，红色表示删除的内容，绿色表示刚刚添加的内容
  • git diff HEAD - 比较工作区与版本库之间的差异
    • HEAD是Git中的一个非常重要的概念，指向当前分支的最新提交节点
  • git diff --cached - 比较暂存区与版本库之间的差异
  • git diff 仓库版本ID1 仓库版本ID2 - 比较两个版本库之间的差异
    • git diff HEAD 仓库版本ID - 比较当前仓库版本与特定仓库版本之间的差异
    • git diff HEAD HEAD~或者git diff HEAD HEAD^ - 比较当前仓库版本与上一个仓库版本之间的差异
    • git diff HEAD HEAD~2或者git diff HEAD HEAD^2 - 比较当前仓库版本与上两个的仓库版本之间的差异，数字2表示指定当前版本的前两个版本，可通过改变数字的指定不同版本
    • git diff HEAD HEAD~ file.txt - 加上文件名后，只会查看这个文件的差异内容

# 从仓库中删除文件

有两种方式

  • 直接删除文件再提交
    • 删除工作区的文件后需要使用git add命令来删除暂存区的文件
  • git rm 文件名 - 工作区和暂存区的文件都会被删除，但还需使用git commit命令提交操作，否则删除的文件在仓库中还存在
    • git rm --cached 文件名 - 删除版本库中的该文件，但不删除工作区的该文件

# 用.gitignore文件忽略文件

.gitignore文件的作用是忽略掉一些不应该被加入到版本库中的文件。

应该忽略哪些文件

  • 一些系统自动生成的文件(工具或软件产生的临时文件)
  • 编译生成的一些中间文件等(例如.class文件，.o文件)
  • 系统运行中自动生成的一些文件(例如日志文件，缓存文件，临时文件)
  • 涉及身份，密码，口令，密钥等包含敏感信息的文件

将文件名写入到.gitignore文件中即可忽略该文件，一行一个文件名，添加到里面的文件要想被忽略有一个前提，就是该文件不能是已经被添加到版本库中的文件。

Git默认是不会将空的文件夹添加到仓库中的。

将文件夹名写入到.gitignore文件中忽略文件夹，文件夹的格式需以斜线结尾(例如temp/)。

.gitignore文件的匹配规则

  • 空行或者以#开头的行会被忽略。#开头表示注释，空行用于可读性的分割
  • 从上到下逐行匹配，每一行表示一个忽略模式
  • 使用标准的Blob模式匹配(简化的正则表达式)
    • *通配符 - 表示匹配任意字符
    • ?限定符 - 匹配单个字符
    • []表示匹配任何一个在列表中的单个字符(例如[abc]，即表示匹配a, b, c其中一个即可)
  • 两个信号**表示匹配任意的中间目录
  • 中括号[]中使用短中线-连接
    • [0-9] - 表示任意一位数字
    • [a-z] - 表示任意一位小写字母
  • !表示取反，当要忽略指令模式以外的文件或目录时即可在模式前加上!号

# 远程仓库(Working with Remotes)

远程仓库地址的两种方式

HTTPS - 当把本地代码推送到远程仓库时需要验证用户名和密码

SSH - 使用SSH协议，把本地代码推送到远程仓库时不需要验证用户名和密码，但需要在Github中添加SSH公钥

  • 在用户根目录中进入.ssh文件夹中，使用ssh-keygen -t rsa -b 4096命令来生成SSH密钥。-t指定协议为RSA，-b指定生成的大小为4096

    • 第一次使用该命令时不需要输入密钥的名称，直接回车，会在.ssh目录下生成一个名为id_rsa的密钥文件。将公钥内容添加到Github即可。

    • 非第一次使用该命令时，可通过输入密钥名称来生成另外一个密钥，生成文件中还包含一个没有扩展名的私钥文件和.pub结尾的公钥文件。将公钥内容添加到Github，并在.ssh目录中创建一个名为config的文件，将下面内容添加到里面。里面的内容表示当访问github.com时，指定使用SSH下的test密钥

本地仓库与远程仓库是两个相互独立的仓库，在本地仓库中做的任何修改都不会影响到远程仓库，同样远程仓库中做的任何修改也都不会影响到本地仓库，故需要一种机制来同步本地仓库和远程仓库的修改内容，让它们的状态保持一致，这就涉及到git pull与git push这两个命令了。

  • git push - 推送，把本地仓库的修改内容推送到远程仓库
  • git pull - 拉取，把远程仓库的修改内容拉取到本地仓库

# 关联本地仓库与远程仓库

git remote add origin(仓库别名) 仓库URL地址 - 添加一个远程仓库，别名为origin

git remote -v - 查看当前仓库所对应的远程仓库的别名和地址

git branch -M main - 指定分支名为main的分支

git push -u origin main - 把本地main分支与别名为origin的远程仓库的main分支关联起来

  • git push -u origin main:main - 具体写法，由于远程仓库分支名与本地仓库分支名相同，故简写
  • 使用了git push命令的-u命令选项后，之后的仓库的push可直接用git push这个简写指令了

git pull origin main - 拉取别名为origin的远程仓库的main分支，把远程仓库的指定分支拉取到本地再进行合并

  • 执行完git pull命令后，git会自动地执行一次合并的操作，如果远程仓库中的修改内容与本地仓库中的修改内容没有冲突，那么合并操作就会成功，否则合并操作就会由于冲突而失败，这个时候就需要手动来解决冲突

从远程仓库获取内容还可使用fetch命令，git pull与fetch命令的区别在于fetch命令只获取远程仓库的修改内容，但是并不会自动地合并到本地仓库中，而是需要手动合并。

# 分支(Branch)

分支是Git中一个非常重要的功能，可以把它看作是代码库中的不同版本，可以独立存在并且有自己的提交记录，分支非常适合团队协作和开发管理，比如多个开发人员可以在自己的分支上进行开发工作，最后再合并到主线代码库中，也可以在一个分支上进行新功能的开发或建立一个问题修复的分支来处理一些bug和缺陷，这样就可以让主线代码仓库处于一个随时可用的比较稳定的状态，而不会影响到其他功能的开发和测试，保证了项目的正常运行和高效协作。

在Git早期的版本中，默认分支的名字是master，由于某些原因在之后的版本中改为了main。

git branch - 查看当前仓库的所有分支

git branch 分支名 - 创建一个新的分支，只是创建并不会自动切换到新的分支

git checkout 分支名 - 切换分支，存在一些潜在的问题，除了切换分支以外，该命令还能用来恢复文件或目录到之前的某一个状态

  • git checkout命令默认操作是切换分支，而不是恢复文件
  • 如果修改了某个文件，即可使用该命令将该文件恢复到之前的状态，假如此时分支名与文件名相同，就会出现歧义

git switch 分支名 - Git 2.23版本才有的新命令，专门用来切换分支

git merge 分支名 - 将不同的分支合并到当前分支中，merge后面的分支名是将要被合并的分支，当前所在的分支就是合并后的目标分支，分支被合并后并不会消失

git log --graph --oneline --decorate --all - 查看分支图与分支ID

git branch -d 分支名 - 删除分支，-d表示删除已经完成合并的分支

  • 如果一个分支已经被合并到其他分支中，那么就可以使用该命令删除这个分支
  • 如果一个分支没有被合并，就不能使用-d参数来删除分支，就需要使用git branch -D 分支名命令来删除，-D表示强制删除

git merge --abort - 中断本次合并

# 解决分支合并冲突

一般情况下，如果两个分支的修改内容没有重合的部分，Git会自动完成合并。如果两个分支修改了同一个文件的同一行代码，Git就不知道应该保留哪个分支的修改内容了，即产生了冲突，就需要手动来解决冲突了。

分支命令也是有一定的规则和含义的，例如feat，是feature的缩写，一般用来表示开发某个功能的分支。

出现冲突后，可使用git status命令来查看冲突文件的列表；也可使用git diff命令查看冲突的具体内容，git会把两个分支的修改内容全部显示出来，并且git会使用左箭头，等号，由箭头分别来表示两个分支的修改内容，其中等号上面的内容表示刚刚main分支中的修改内容，等号下面到右箭头之间的部分是刚刚在当前分支中添加的内容。解决该冲突需要做的就是手动编辑该冲突文件，留下想要的内容之后，然后再重新提交。

# 变基(rebase)与回退

除了git merge命令可合并不同的分支外，git rebase命令也可以用来合并不同的分支。

git rebase 分支名 - 可以在任意分支上执行rebase操作

  • 在子分支上执行 - git rebase 主分支名
  • 在主分支上执行 - git rebase 子分支名

在Git中，每个分支都有一个指针(HEAD)，执行当前分支的最新提交记录，在执行rebase操作时，Git会先找到当前分支和目标分支的共同祖先，再把当前分支上从共同祖先到最新提交记录的所有提交都移动到目标分支的最新提交后面，即从分叉点将整个分支都移动到目标分支的最新提交记录后面。

git checkout -b 分支名 分支ID - 恢复到分支某一时间点的状态

Rebase和Merge合并分支命令有什么区别？如何区分使用？

git merge

  • 优点 - 不会破坏原分支的提交历史，所有提交记录与合并历史都会保留下来，方便回溯和查看
  • 缺点 - 会产生额外的提交记录和两条分支线的合并，使得提交记录历史变得复杂

git rebase

  • 优点 - 不需要新增额外的提交记录到目标分支，可以形成一个线性的提交历史记录，非常直观和干净
  • 缺点 - 会改变提交历史，需避免在一个共享分支上进行rebase操作，在共享分支上进行rebase操作会造成一些困扰给开发的同事，一般不会在公共的分支上执行rebase操作

如果只想把两个分支合并，不关心提交历史，就可以使用git merge命令。

如果是个人开发的分支并且希望提交历史更加清晰，就可以使用git rebase命令。

# Git分支工作流模型

工作流模型就是比较好的规范和流程。

# GitFlow模型

将分支分成了五种类型，每种类型都有各自的用途。

  • 主线分支(main分支/master分支) - 也叫基线分支，是项目的核心分支，包含了项目的最新稳定版本的代码
    • 应该随时保证主线分支中的代码时可发布的，一般来说主线分支的代码会被部署到生产环境中
    • 主线分支中的代码是不允许直接修改的，只能通过合并分支的方式来修改，每次合并分支都建议生成一个新的版本号，这样可以方便追踪和回溯
    • 通过git tag命令来标记版本号，版本号的规则为
      • 主版本(Major Version) - 主要的功能变化或重大更新
      • 次版本(Minor Version) - 一些新的功能，改进和更新，通常不会影响现有功能
      • 修订版本(Patch Version) - 一些小的bug修复，安全漏洞补丁等。通常不会更改现有功能和接口
      • git tag -a v0.0.1 - 创建标签v0.0.1，之后自动进入vim编辑器进行注释编写
      • git tag -a v0.0.1 -m "msg" - -m指定注释
      • git tag -d v0.0.1 - 删除本地标签，之后使用命令git push -u origin main --delete v0.0.1
  • 问题修复分支 - 包含了项目某个问题修复的源码，用于修复线上的问题
    • 一般来说问题修复分支是从主线分支中分离出来的，修复完成后会合并到主分支和开发分支中，当问题修复分支合并完成之后，一般会将这个问题修复分支删除掉
    • 问题修复分支一般也会更新项目的小版本号
  • 开发分支(dev分支) - 是从主线分支中分离出来的，包含了项目的最新开发版本的代码，用于开发和测试
    • 开发分支也是项目的核心分支，与main分支一样，是一个长期存在的分支
  • 功能分支(feat分支) - 一般是指包含了项目某个新功能的代码，用于开发新的功能，都是从开发分支中分离出来的，当功能分支的代码稳定后，会以某种方式合并回开发分支中
    • 例如feature-login-page分支 - 是一个功能分支，用于开发登录页面的功能，该功能分支中应该只包含登录页面相关的功能代码，等到开发完成后，合并回开发分支中，然后再按照版本发布的计划执行版本发布的流程
  • 版本发布分支(release分支) - 也叫预发布分支，包含了项目最新预发布版本的代码，用于发布前的测试和验证
    • 一般来说预发布分支是从开发分支中分离出来的，当预发布分支的代码稳定后，会合并到主分支和开发分支中
    • 在测试和验证的过程中发现的问题会在预发布分支中进行修改，提交，并且cherry-pick到开发分支中。当预发布分支的代码稳定后，会合并到主分支和开发分支中，然后会将预发布分支删除掉

这些不同种类的分支中，按照生命周期的不同又可以分成两种类型，即主要分支和辅助分支

  • 主要分支 - 主分支和开发分支，这两个分支是项目的核心分支，一般都是长期存在的
  • 辅助分支 - 问题修复分支，功能分支以及版本发布分支，这几个分支在完成各自的功能之后就可以删除掉了

GitFlow模型适用于团队技术水平适中，有一定的开发流程和规范的团队。

# GitHubFlow模型

GitHubFlow模型则适用于一些技术水平比较高的团队或开源项目。

GitHubFlow模型只有一个长期存在的主分支，并且主分支上的代码是可以直接部署到生产环境中的，一般会设置分支保护，禁止团队成员直接在主分支上进行提交，团队成员们可以从主分支中，分离出自己的分支进行开发和测试，然后在本地分支提交代码。等到开发完成之后，可以发起一个Pull Request - PR，即拉请求或合并请求，团队成员可以对代码进行Review评审，如果没有问题就可以将这个PR发布并合并到主分支中。GitLab中也有类似的功能，叫做MergeRequest - MR。

# 相关规范

分支的命名规范

  • 推荐使用具有意义的描述性名称来命名分支
  • 版本发布应该根据版本号命令，例如v1.0.0
  • 功能性分支应该根据其功能或任务来命名
  • 修补补丁和错误应该根据其问题编号命名

分支的创建规范

  • 定期合并已成功验证的分支，并及时删除已经合并的分支
  • 保持合适的分支数量，确保及时将功能性分支合并回主分支
  • 为分支设置合适的管理权限
