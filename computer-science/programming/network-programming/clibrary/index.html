C/C++静态库与动态库的制作和使用

# 概述

  • 程序代码编写过程中通常把自定义函数和类从主程序中分离，函数和类的声明写在头文件中，主程序中要包含头文件，编译时要和程序文件一起编译

  • 在实际开发中，处于技术保密或其它方面考虑，通常不提供公用函数库的源代码

  • 当链接时既有静态库又有动态库，会优先使用动态库

  • 静态库后缀为.a

动态链接和静态链接是什么？它们之间的区别是什么？

主要区别在于链接到程序的时间和方式

静态链接

  • 在编译期间，静态库的代码就被包含进了目标可执行文件中，之后即使没有静态库文件，也能正常运行
  • 生成的可执行文件较大，因为它包含了所有需要的库函数
  • 更改静态库的功能或修复错误需要重新编译应用程序，分发新版本的可执行文件

动态链接

  • 动态链接是在运行时完成的，也就是说，在编译阶段并不会把库文件的代码加入到可执行文件，而只是添加一些引用信息。在程序执行时，由操作系统负责将动态库加载到内存中供程序使用
  • 可执行文件较小，因为它仅仅包含对动态库的引用，而非实际的代码
  • 如果多个应用程序都使用相同的库，那么这个库只需要在内存中保留一份即可，节约了系统资源
  • 当动态库更新时，不需要重新编译和链接应用程序，运行程序时自然会调用新版本的库

每种链接方式都有其利弊。静态链接产生的程序更独立，但可能导致二进制文件很大；动态链接可以共享库提高效率，但如果相关的库不在系统上，那么程序无法运行。

标准库函数和系统调用的区别？

来源

  • 系统调用 - 这些函数来自操作系统内核。它们是操作系统提供给应用程序直接访问硬件和系统资源的基础界面，例如读写文件、发送网络数据、创建进程等
  • 标准库函数 - 这些函数是由C语言(或其他语言)运行时环境提供的，例如printf、strcpy、malloc等

实现

  • 系统调用 - 由操作系统内核代码实现，当一个进程执行系统调用时，它需要切换到内核模式来运行特权代码
  • 标准库函数 - 通常使用用户模式下的普通代码实现，并且可能在其内部使用系统调用以提供其功能

性能

  • 系统调用 - 因为涉及用户空间到内核空间的上下文切换，所以相比于标准库函数，系统调用通常会有更高的开销
  • 标准库函数 - 也可能引入一定的性能开销，如果它们内部使用了系统调用，但如果只是在用户空间进行计算，那么它们的开销就会小得多

可移植性

  • 系统调用 - 通常依赖于特定的操作系统，所以在不同操作系统之间的可移植性较差
  • 标准库函数 - 大多数情况下，标准库函数在各种平台上的行为都是一致的，所以具有更好的可移植性
Linux的共享库(Shared Library)？

在Linux系统中，共享库是一种可执行代码和数据的模块，可以在多个程序之间共享。共享库的使用有助于减小可执行文件的大小，提高程序的重用性，以及便于软件更新和维护。

gcc的-shared参数表示生成共享库。

静态库和动态库是什么？优缺点是什么？使用场景？

静态库和动态库是两种不同形式的软件库，用于封装和共享代码、函数以及数据结构，以供程序在运行时调用。

静态库

  • 特点

    • 静态库是在编译时被链接到程序中的，链接器会把库的代码和数据复制到可执行文件中

    • 在程序运行时，静态库的代码不再发生变化，它的内容被完全复制到每个使用它的可执行文件中

  • 文件格式

    • 静态库通常以.lib(在Windows上)或.a(在Unix/Linux上)为扩展名。

    • 在Windows上也可能包含.dll(动态链接库)和.lib(导入库)两个文件，但这是为了支持导入库的链接方式

  • 优点

    • 静态库的优点之一是独立性高，不依赖于运行时环境，使得程序的分发和部署相对简单

    • 由于整个库的代码被复制到可执行文件中，可以减少对外部依赖

  • 缺点

    • 如果多个可执行文件使用同一个静态库，每个可执行文件都会包含一份库的拷贝，可能导致可执行文件体积较大

    • 静态库的更新需要重新编译程序

动态库

  • 特点

    • 动态库是在运行时被加载到内存中的，程序在运行时通过操作系统的动态链接器动态地加载和链接库

    • 动态库的代码和数据不会被复制到可执行文件中，而是保留在库文件本身

  • 文件格式

    • 动态库通常以.dll(在Windows上)或.so(在Unix/Linux上)为扩展名

  • 优点

    • 动态库的主要优点之一是节省内存，因为库的代码和数据只需加载一次，即使被多个程序使用

    • 动态库的更新不需要重新编译程序，只需要替换动态库文件即可

  • 缺点

    • 动态库对运行时环境有一定的依赖，如果系统中缺少相应的库，程序可能无法运行

    • 动态库在运行时需要进行额外的加载和链接工作，可能引入一些性能开销

使用场景

  • 静态库
    • 适用于小型项目或独立的可执行文件
    • 依赖独立性高，部署简单
  • 动态库
    • 适用于多个程序共享同一份库的情况，以减少内存占用
    • 适用于大型项目，可以在运行时更新库而无需重新编译
    • 支持插件系统，允许在不停止程序的情况下加载新功能

# 静态库(Static Library)编译过程

  • 静态库在编译的时候，主程序文件和静态库文件一起编译，把主程序与主程序中用到的库函数一起整合进了目标文件，这样即可在编译后使可执行程序独立运行
  • 但，当所使用的源代码或静态库发生变化时，程序必须重新编译才能生效

# 动态库(Dynamic Library)编译过程

  • 动态库在编译时，不会被链接到目标代码中，而是在程序运行时才会被载入，并且在程序运行时还需指明动态库的目录

  • 当库文件发生变化时，只需将发生变化的库文件重新编译即可，程序不需要重新编译

  • 动态库编译后名字一般都包含lib字符，其后面的字符才为库名

  • 动态库的后缀为.so

  • 在使用动态库模式编译程序时，并不会把库文件内容整合到程序中，编译后，只有程序使用时，库文件才会被加载到程序当中。因此，使用时需要设置一个环境变量LD_LIBRARY_PATH，用于指定动态库的搜索目录

# 补充

  • C语言标准库函数目录 - /usr/lib/gcc/x86_64-redhat-linux/版本号
  • 可以改变库函数文件
  • 程序一般会提供头文件和库文件，通过查看库文件可看函数接口
