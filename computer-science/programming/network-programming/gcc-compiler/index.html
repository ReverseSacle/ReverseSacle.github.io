# 概述

  • gcc - 用于编译C语言(.c结尾文件)
  • g++ - 用于编译C++(.cpp/.cc结尾文件)
什么是编译器，都有哪些`C++`编译器，区别是什么？

编译器是一种将高级编程语言源代码翻译成目标机器代码或可执行文件的工具。其主要任务是将程序员编写的高级语言代码转换成计算机硬件能够理解和执行的低级机器代码。编译器通常包括预处理器、编译器、汇编器和链接器等组件。

  • GNU Compiler Collection (GCC)

    • GCC是一个开源的编译器套件，支持多种编程语言，包括C++

    • 它是许多Unix/Linux系统的默认编译器，也在其他平台上广泛使用

    • 具有强大的优化功能和广泛的平台支持

  • Clang

    • Clang 是一个由LLVM项目提供的C++编译器前端

    • 它设计注重速度和模块化，可以作为独立的前端或与LLVM后端配合使用

    • Clang的静态分析工具也使其在代码质量分析方面很有优势

  • Microsoft Visual C++ Compiler

    • Microsoft Visual C++是Windows平台上的官方C++编译器

    • 集成到Visual Studio开发环境中，提供了对Windows特性的广泛支持

  • Intel C++ Compiler

    • Intel提供了用于多种平台的C++编译器，特别注重性能和并行计算

    • 支持许多优化技术，如向量化、多线程等

  • IBM XL C/C++ Compiler - IBM的XL C/C++ Compiler是针对IBM Power 和 System z体系结构的编译器，提供了对IBM硬件的优化

这些编译器之间的区别主要包括以下几个方面

  • 性能优化 - 不同编译器可能在性能优化方面有不同的策略和实现
  • 平台支持 - 不同编译器可能对不同的平台有更好的支持
  • 标准遵循 - 编译器可能在对C++标准的支持上有差异，包括C++98、C++11、C++14等版本
  • 工具和附加功能 - 编译器提供的附加工具、静态分析和调试支持等方面可能有所不同
VS中的DeBug和Realease区别？(*)

在Visual Studio中，Debug(调试)和Release(发布)是两种不同的构建配置，用于指定在不同情况下生成可执行程序的方式。这两种配置主要针对开发和发布两个不同的阶段。

  • 优化级别

    • Debug - 使用较低级别的优化，以便在调试时更容易观察变量和执行流程。通常包含符号信息，以支持源代码级的调试。编译速度较快

    • Release - 使用较高级别的优化，以提高程序的执行性能。通常不包含符号信息，使得生成的可执行文件较小。编译速度较慢

  • 符号信息

    • Debug - 包含详细的符号信息，使得在调试器中能够进行源代码级的调试。生成的可执行文件较大

    • Release - 通常不包含详细的符号信息，以减小可执行文件的大小。这样做可以加速程序的加载和执行

  • 运行时检查

    • Debug - 启用一些运行时检查机制，例如数组越界检查和空指针检查，以提高程序的健壮性

    • Release - 关闭运行时检查，以减小程序的运行时开销

  • 内联函数

    • Debug - 通常不启用函数内联，以方便调试时跟踪函数的调用栈

    • Release - 启用函数内联，以提高程序的执行效率

  • 编译器优化

    • Debug - 采用较少的编译器优化，以减小生成代码的复杂性，方便调试

    • Release - 使用更多的编译器优化，以提高生成代码的执行效率

  • 异常处理

    • Debug - 通常启用详细的异常信息和调试信息，以方便调试时追踪异常

    • Release - 可能减少异常处理的信息，以提高程序执行效率

  • 预处理器定义

    • Debug - 通常定义了一些预处理器符号，例如_DEBUG，以便在代码中包含或排除特定的调试代码

    • Release - 可能定义一些与发布相关的符号，例如NDEBUG，以排除与性能无关的调试代码

在Visual Studio中，可以通过在解决方案配置管理器中选择Active Solution Configuration来切换Debug和Release配置。通常，在开发和调试阶段使用Debug配置，在发布最终版本时使用Release配置。

# 安装

  • yum -y install gcc
  • yum -y install gcc-c++
  • 网速慢时，执行以下命令(错误就终止再执行以下命令直到指令正常完成)
    • yum clean all
    • yum makecache
  • gcc --version - 查看gcc版本

# 编译命令

  • gcc -W c文件 - 编译过程中提示错误警告(-W)

    • gcc -Wall c文件 - 将警告黏在一起(-Wall)，可防止潜在错误

  • gcc -Wall c文件 -o 文件名 - -o指定输出文件名或文件路径

  • gcc -v -Wall c文件 - -v指令用于显示详细的依赖和环境等情况

  • gcc -Wcomment c文件 - -Wcomment指令用于检测注释是否嵌套

  • gcc -Wformat c文件 - -Wformat指令用于检测变量格式的使用是否错误
  • gcc -Wunused c文件 - -Wunused指令用于检测没有被使用的声明
  • gcc -Wimplicit c文件 - -Wimplicit指令用于检测被使用的函数没有声明
  • gcc -Wreturn-type c文件 - -Wreturn-type指令用于检测被使用的函数返回类型是否错误
  • gcc -Wconversion c文件
  • gcc -Wshadow c文件
  • gcc -Wcast-qual c文件
  • gcc -Wwrite-strings c文件
  • gcc -Wtraditional c文件

# 代码分层管理编译

  • gcc -Wall c文件1 c分支文件2 -o 文件名 - 将主程序代码分割成xx.h，xx.c，main.c时使用，.h头文件不需要编译

  • 预编译，生成.o目标文件，不可执行

    • gcc -c c文件 - 生成可链接目标文件

  • gcc xx.o xx.o -o 文件名 - 链接.o目标文件生成可执行文件

  • 目标文件链接顺序，gcc不会出现下面情况

    • 某些编译器，链接时被调用的目标文件要放到前面，调用的目标文件放到后面

# 编译时链接外部库

  • gcc -Wall main.c /usr/lib/libm.a -o clac - 指定外部库路径
  • gcc -Wall main.c -lm -o calc - -lm表示将指定库路径当成系统标准库目录
  • gcc -Wall -I库所在路径 -o 文件名 - 指定非标准库的索引路径
  • gcc -Wall -L库所在路径 -l库名 -o 文件名 - 指定库路径链接
  • 目标文件链接顺序，gcc不会出现下面情况
    • 某些编译器，链接时被调用的库路径要放到调用者前面

# 目标文件转成库文件

静态库(static libraries) - .lib结尾(windows)，.a结尾(Linux)，多少库代码被调用就拷贝多少库代码

动态库(shared libraries) - .dll结尾(windows)，.so结尾(Linux)，库代码被调用时会生成指向库代码的指针，将所有指针以制表的方式整合，当库被调用时通过表中指针索引并调用库实例中的代码，减少内存浪费

# GCC编译默认以动态库的方式

  • 通过gcc -c c文件编译成.o目标文件

  • ar rcs 文件名.a 目标文件1名.o 目标文件2名.o - 将目标文件黏合成库文件，不包含main.o的目标文件

    • r表示替换(replace)，它告诉 ar 命令如果已经存在同名的文件，则替换它
    • c表示创建归档文件，如果指定的归档文件不存在，则创建一个新的归档文件
    • s表示创建符号表(symbol table)，它会在归档文件中包含用于链接的符号表信息

  • 链接

    • gcc main文件名.c 库名.a -o 文件名 - 外部库文件的链接
    • cp xxx.a 系统标准库路径 - 将外部库文件拷贝到系统标准库路径
    • gcc -Wall main文件名.c -lxxx -0 文件名 - 将之前拷贝到系统标准库路径的库通过-l库名指令链接(不推荐)
    • gcc -Wall -L. -l库名 -o 文件名 - .表示当前工作目录(推荐做法)
    • gcc -Wall -L/etc/lib -l库名 -o 文件名 - 指定库路径链接(推荐做法)

# 指定C标准编译

GCC默认以GNU C标准编译

  • gcc -Wall -ansi main.c -o 文件名 - 以ANSI标准编译(符合模式)
  • gcc -Wall -ansi -pedantic -o 文件名 - 以ANSI标准编译(严格模式)
  • gcc -Wall -std=c99 main.c -o 文件名 - 以C99标准编译(符合模式)
  • gcc -Wall -ansi -D_GNU_SOURCE main.c -o 文件名 - 以ANSI标准编译(符合模式)，但可以调用GNU C标准的库

# 预处理

GNU C preprocessor(简称cpp)预处理器

  • 宏定义

  • gcc -Wall -DTEST dtest.c -o 文件 - -D指令用于宏定义

  • gcc -Wall -DNUM=123 main.c -o 文件 - 指定变量值(默认为1)，替换NUM

  • gcc -E c文件 - -E指令用于显示预处理器的结果

  • gcc -Wall -c -save-emps hello.c - 生成.i结尾的预处理结果文件

  • gcc -Wall -g c文件 - -g指令用于将调试信息添加到目标文件和可执行文件中

# 编译时(-O)代码优化

time 可执行文件 - 查看执行时间

gcc -OLEVEL c源文件 - 自动优化代码，LEVEL值范围0, 1, 2, 3，值越大编译时间越长，并不是优化等级越高，优化效果越好。

  • 0 - 不优化
  • 1 - 低级别优化
    • 启用了一些基本的优化，如循环展开和函数内联
    • 试图减少指令数和改善代码布局
  • 2 - 投入生产的优化方案
    • 在O1优化的基础上
    • 进一步的指令调度和循环变换
    • 更强大的内联
    • 更多的标准性能优化
  • 3 - 高级别优化
    • 包含所有-O2级别的优化
    • 更激进的优化，可能会导致编译时间增加
    • 针对特定架构的优化

# 编译原理

  • 预处理(Preprocessing) - 调用(cpp预处理器)扩展宏
    • cpp xxx.c > xxx.i - C文件的预处理命令，.i文件中的宏都被扩展了
    • cpp xxx.cc > xxx.ii - C++文件的预处理命令
  • 编译(Compilation) - 将C文件编译为汇编语言
    • gcc -S xxx.i - -S表示声明不生成目标文件，而是生成汇编语言的代码文件(.s结尾)
  • 汇编(Assembly) - 将汇编语言编译为机器码
    • as xxx.s -o xxx.o - 用汇编文件生成目标文件
  • 链接(Linking) - 创建最终可执行文件
    • gcc xxx.o - 生成可执行文件
C++从源代码到可执行程序的流程是怎样的？(C++中程序从编写到运行的过程？)

  • 编写源代码 - 开发者使用文本编辑器或集成开发环境(IDE)编写C++源代码。C++源代码文件通常以.cpp或.cc为扩展名

  • 预处理

    • 预处理阶段处理源代码中的预处理指令，例如#include、#define等

    • 预处理器会替换宏、展开头文件等，生成一个被处理过的源代码文件

  • 编译

    • 编译器将预处理后的源代码翻译成汇编语言(Assembly)

    • 汇编语言是与机器体系结构相关的低级代码

  • 汇编 - 汇编器将汇编语言代码翻译成机器码(二进制代码)，生成目标文件(通常以.o或.obj为扩展名)

  • 链接

    • 链接器将目标文件与库文件(如果有的话)和其他目标文件链接在一起，创建最终的可执行文件

    • 链接器解析符号引用，解决函数和变量的地址

  • 库链接

    • 如果程序引用了外部库，链接器会将这些库链接到可执行文件中

    • 静态链接会将整个库的代码和数据复制到可执行文件中，而动态链接则在运行时加载所需的库

  • 可执行文件生成

    • 最终生成的可执行文件包含了机器码、数据段、符号表等信息

    • 在Unix/Linux系统中，可执行文件通常没有扩展名，而在Windows系统中，可执行文件通常以.exe为扩展名

  • 运行程序 - 用户运行生成的可执行文件，操作系统加载程序到内存，并开始执行

动态编译与静态编译是什么？优缺点？使用场景？

动态编译和静态编译是两种不同的编译方式，它们主要指的是在软件开发中编译过程中代码与库的链接方式的不同。

静态编译(Static Compilation)

静态编译是指在程序源代码被编译成机器码(或中间代码)的过程中，将所有的依赖库和模块都链接到最终的可执行文件中。在这个过程中，链接器会将源代码中引用的外部函数和库的代码嵌入到最终的可执行文件中。这样，生成的可执行文件在运行时就不再需要外部的依赖，它包含了程序的全部代码和数据。

  • 特点

    • 在静态编译中，编译器将源代码翻译成目标机器代码，并将所有需要的库和模块的代码都链接到最终的可执行文件中

    • 所有的代码都在编译阶段被确定，生成的可执行文件是一个独立的、自包含的二进制文件

  • 优点

    • 生成的可执行文件相对独立，不依赖于外部的库，可以轻松分发和部署

    • 可以在编译时进行各种优化，生成高效的机器代码

  • 缺点

    • 每次更新代码或库时都需要重新编译整个程序

    • 生成的可执行文件可能较大

动态编译(Dynamic Compilation)

动态编译是指在程序运行时，将源代码翻译成机器码(或中间代码)的过程。这种方式不会在编译阶段将所有的依赖库和模块都链接到可执行文件中。相反，程序在运行时会动态加载外部依赖的库和模块。动态编译一般涉及到解释器或即时编译器(JIT Compiler)，它们会将程序的部分代码在运行时翻译成机器码。

  • 特点

    • 在动态编译中，编译器将源代码翻译成中间代码或字节码，而不是直接生成目标机器代码

    • 在运行时，中间代码由解释器或即时编译器(Just-In-Time Compiler，JIT Compiler)翻译成目标机器代码

  • 优点

    • 允许在运行时动态加载和执行代码，支持动态更新和扩展

    • 可以实现更高级的优化，因为编译过程可以考虑到运行时的上下文

  • 缺点

    • 需要解释器或即时编译器的支持，可能引入一些性能开销

    • 生成的可执行文件通常依赖于一个运行时环境，需要确保环境的可用性

使用场景

  • 静态编译
    • 适用于要求独立性和可分发性的场景，如嵌入式系统、可执行文件的发布等
    • 适用于对性能有较高要求的场景，因为静态编译的程序在运行时无需进行额外的解释和编译
  • 动态编译
    • 适用于需要支持动态更新和扩展的场景，如插件系统、脚本语言解释器等
    • 适用于需要更高级别优化的场景，因为编译可以在运行时考虑到实际上下文

# 用于辅助的Linux命令

  • file 文件名，显示以下内容
    • ELF - 全称Executable and Linking Format，表示该文件为可执行文件
    • LSB - 全称Least Significant Byte，表示字节顺序为低位在前；MSB表示字节顺序为高位在前
    • not stripped - 表示文件包含符号表(Symbol Table)
  • nm 文件名 - 查看文件符号表
    • T 符号 - 表示函数已被定义
    • U符号 - 表示函数没有被定义，当使用动态库时会存在U符号
  • ldd 可执行程序 - 查看程序依赖的动态库

# 用于辅助的工具

The GNU profiler gprof

  • 记录程序的执行情况，如函数的调用次数，代码各个地方的执行时间等

  • 命令

    • gcc -pg c文件 - 生成gmon.out文件
    • gprof c可执行文件 - 读取gmon.out文件并显示C可执行文件的执行信息

The GNU coverage testing tool gcov

  • 统计程序里每一行的执行时间，用于查找代码中哪一部分没有被使用
  • 命令
    • gcc -fprofile-arcs -ftest-coverage c文件 - 生成cov.c文件和cov.gcno文件，执行C可执行程序后又会生成cov.gcda文件
    • gcov cov.c - 生成cov.c.gcov文件
    • vi cov.c.gcov - 查看，第一行表示执行次数(五个#表示没有执行过)
    • grep "#####" *.gcov - 定位没有执行过的地方
